# Предложение: архитектура подсистемы отчетов

## Цели

1. **Изоляция отчетов**: новый отчет добавляется как отдельный модуль без правок ядра UI/сервисов.
2. **Общая инфраструктура**: не дублировать форму параметров, валидацию, выбор периодов, поиск цен, экспорт.
3. **Единый UX**: пользователь выбирает отчет в меню, вводит параметры в едином окне, видит результат и может сохранить.

## Ключевая идея

Сделать подсистему отчетов из трех слоев:

- **Report Plugin API** — контракт отдельного отчета.
- **Report Runtime** — общий движок: загрузка плагинов, сбор параметров, выполнение, рендер/экспорт.
- **Common Report Components** — переиспользуемые модели параметров и helper-сервисы.

## 1) Контракт отчета (плагина)

Каждый отчет реализует интерфейс `ReportDefinition`:

- `id: str` — стабильный идентификатор (`cashflow.by_month`).
- `title_key: str` — ключ локализации для меню/заголовка.
- `description_key: str` — ключ описания.
- `parameter_schema() -> ReportParameterSchema` — декларативная схема параметров.
- `build_plan(params, context) -> ReportExecutionPlan` — как получить данные.
- `render(data, params, context) -> ReportResult` — подготовка табличного/текстового/графического вывода.

Важно: базовый сценарий — отчет объявляет параметры и логику, а стандартный runtime строит форму/экспорт.
Но для сложных случаев отчет может предоставить **собственный UI/renderer/exporter** через extension points (см. ниже),
чтобы ядро не содержало код под все возможные форматы.

## 2) Общая модель параметров (без дублирования)

Ввести универсальные типы полей:

- `AccountSetField` (один/несколько наборов счетов, включая сохраненные пресеты).
- `PeriodField` (общий период + режим разбивки).
- `PriceRulesField` (разрешенные источники, окно времени, fallback-правила).
- `CurrencyField` (валюта отчета, опционально "авто").
- `EnumField`, `BoolField`, `NumberField`, `TextField`.

`ReportParameterSchema` содержит:

- список полей;
- дефолты;
- правила валидации (`required`, "период не пустой", "окно цен > 0", и т.д.);
- условную видимость/доступность (например, поле окон цен только если включена конвертация).

Это позволит большинству отчетов только объявлять, **какие** параметры нужны, а не писать UI вручную.
При этом схема должна поддерживать кастомные поля:

- `CustomField(type_id, payload)` — поле, для которого конкретный отчет регистрирует `FieldEditorFactory`.
- `ValidationHook` — отчетный callback для нестандартной валидации (межполевые и доменные проверки).

Таким образом, нестандартные входные параметры поддерживаются без раздувания ядра.

## 3) Разбивка периода как отдельная стратегия (на базе `dateutil.relativedelta`)

Вместо custom-логики в каждом отчете:

- `PeriodRange` — общий интервал (`from`, `to`).
- `PeriodSplitStrategy`:
  - `none`
  - `daily`
  - `weekly`
  - `monthly`
  - `quarterly`
  - `yearly`
  - `custom_relative_delta` (с параметром `relativedelta`)

Общий helper `split_period(range, strategy)` отдает список "бакетов". Любой отчет использует это одинаково.

Почему не только `timedelta`, а `relativedelta`:

- `timedelta(30d)` не равен календарному месяцу.
- Календарные кварталы/годы зависят от границ дат, а не от фиксированного количества секунд.
- Пользователь обычно мыслит категориями "по месяцам/кварталам", а не "каждые N секунд".

Итог: базовым примитивом для разбивки лучше сделать `relativedelta`, потому что он естественно выражает календарные шаги.

## 4) Переиспользование существующего `PriceService`

Отдельный `ReportPricingService` не нужен. Отчеты используют текущий `PriceService`.

- принимает `PriceRules`;
- вызывает `PriceService.get_price(...)` для фактического поиска;
- при необходимости расширяем сам `PriceService` (например batch API, диагностику, кэш), чтобы не плодить второй слой с похожей ответственностью.

Что уже умеет `PriceService`:

- фильтры по окну времени и источникам;
- поиск цепочки кросс-курсов;
- выбор лучшего пути по количеству hops и близости дат.

Если report-слою чего-то не хватает, добавляем это в `PriceService`:

- единый контракт для массовых запросов внутри отчета;
- отчетные предупреждения/трассировка для показа пользователю;
- отчетно-локальный кэш (чтобы не делать повторные одинаковые вызовы).

То есть источник правды по ценам один — `PriceService`.

## 5) Runtime/оркестратор

`ReportManager`:

- обнаруживает `ReportDefinition` (регистрация в реестре или динамическая загрузка из `app/reports/plugins`).
- отдает список отчетов для меню.
- по выбранному отчету открывает `ReportParameterDialogBuilder`.
- запускает выполнение через `ReportRunner` (в будущем можно вынести в фон).
- маршрутизирует в кастомный UI/renderer/exporter, если отчет их зарегистрировал.

`ReportRunner`:

1. валидирует параметры;
2. собирает `ReportContext` (repos/services/settings/translator);
3. вызывает `build_plan`/`render`;
4. возвращает `ReportResult`.

## 6) Результат отчета: сначала файл, потом просмотр

`ReportResult` как единый контейнер метаданных генерации:

- `output_file_path` (основной сгенерированный файл)
- `mime_type`/`format`
- `warnings[]` (например, цены не найдены)
- `open_hint` (как открыть системным приложением)

UI-слой умеет:

- запустить генерацию в файл;
- после успеха открыть файл стандартным системным приложением;
- показать путь и предупреждения.

Для нестандартных форматов:

- отчет выбирает `OutputWriter` (CSV/XLSX/HTML/PDF/любой кастомный writer);
- runtime обеспечивает единый lifecycle записи и открытия файла.

Тогда ядро предоставляет только extension points и жизненный цикл, а специфичный формат остается внутри отчета.

## 7) Встраивание в текущее приложение

1. Добавить пакет `app/reports/`:
   - `contracts.py`
   - `schemas.py`
   - `manager.py`
   - `runner.py`
   - `dialogs/parameter_dialog.py`
   - `renderers/`
   - `plugins/`

2. В `MainWindow` добавить меню **Reports**:
   - список отчетов из `ReportManager.list_definitions()`
   - action "Generate..." для каждого отчета

3. Контекст отчета собрать из уже существующих репозиториев и сервисов (`AccountRepo`, `TransactionRepo`, `PriceRepo`, `BalanceService`, `PriceService`, `AppSettings`).
4. В `AppSettings` добавить путь каталога для генерации отчетов (`reports_output_dir`) и использовать его как дефолт при запуске отчета.

## 8) Пример жизненного цикла (как видит пользователь)

1. Пользователь открывает **Reports → Cash Flow by Month**.
2. Runtime строит форму из `parameter_schema()`:
   - наборы счетов
   - период
   - разбивка (месяц)
   - правила поиска цен
   - валюта отчета
3. После `Generate` отчет сразу пишет файл в каталог из настроек.
4. Приложение открывает файл системным приложением для соответствующего типа (и при необходимости показывает путь/предупреждения).

## 9) Как добавить новый отчет (целевая DX)

Разработчик делает один файл в `app/reports/plugins/`, например `net_worth_trend.py`:

1. класс `NetWorthTrendReport(ReportDefinition)`;
2. объявляет `parameter_schema`;
3. пишет `build_plan` + `render`.

Никаких изменений в `MainWindow`, базовых диалогах и экспортерах — отчет подхватывается реестром.

## 10) Минимальный roadmap внедрения

### Этап 1 (MVP)

- Реестр отчетов (ручная регистрация).
- 1 общий диалог параметров по схеме.
- 1-2 отчета (например, cash flow по месяцам и баланс на даты).
- Экспорт CSV.

### Этап 2

- Диагностика поиска цен + предупреждения в UI.
- Пресеты параметров отчета (сохранить/загрузить).
- Фоновое выполнение долгих отчетов.

### Этап 3

- Авто-discovery плагинов.
- Доп. экспорты (XLSX/HTML).
- Графики/визуализация.

## 11) Почему это отвечает требованиям

- **Независимость**: каждый отчет — отдельный модуль по контракту.
- **Переиспользование**: общие поля параметров, валидация, разбиение периодов, поиск цен, экспорт.
- **Стабильное ядро**: UI и orchestration меняются редко; новые отчеты добавляются без правки core.
